# JuiceFS preadv/pwritev 测试套件

本测试套件用于验证 JuiceFS 文件系统对 `preadv`/`pwritev` 及其扩展版本 `preadv2`/`pwritev2` 系统调用的支持情况。

## 环境要求

- Linux 操作系统（内核版本建议 4.0+ 以支持完整的 preadv2/pwritev2 功能）
- GCC 编译器
- 已挂载的 JuiceFS 文件系统（作为测试目录）

## 构建与运行

### 构建测试程序

```bash
cd test/preadv_test
make
```

### 运行完整测试套件

```bash
./run_tests.sh /path/to/juicefs/mountpoint
```

### 运行单个测试程序

```bash
./test_basic /path/to/juicefs/mountpoint
./test_flags /path/to/juicefs/mountpoint
./test_odirect /path/to/juicefs/mountpoint
./test_advanced /path/to/juicefs/mountpoint
```

### 清理构建产物

```bash
make clean
```

## 测试架构

### 公共模块 (common.h)

所有测试程序共享公共头文件，提供：

- **常量定义**:
  - `BLOCK_SIZE`: 4096 字节（标准块大小）
  - `TEST_FILE_SIZE`: 256KB（测试文件大小）

- **宏定义**:
  - `PASS(name)`: 标记测试通过并输出信息
  - `FAIL(name, ...)`: 标记测试失败并输出详细信息
  - `SKIP(name, ...)`: 标记测试跳过（功能不支持时）

- **工具函数**:
  - `create_test_file()`: 创建指定大小的测试文件，填充可验证的模式数据
  - `get_time_sec()`: 获取当前时间（秒），用于性能测试
  - `print_summary()`: 输出测试结果统计摘要

### 测试结果判定

每个测试用例返回三种状态之一：

| 状态 | 含义 | 说明 |
|------|------|------|
| `[PASS]` | 通过 | 功能正常工作 |
| `[FAIL]` | 失败 | 功能异常或数据不一致 |
| `[SKIP]` | 跳过 | 当前环境不支持该功能（非错误） |

## 测试用例详解

### 1. 基础测试 (test_basic)

**目标**: 验证 `preadv`/`pwritev` 的基本功能和正确性

**测试数量**: 10 个

#### 1.1 test_preadv_basic - 基础向量读取

**描述**: 使用 3 个 iovec 结构体从文件起始位置读取 4096 字节数据

**测试步骤**:
1. 创建 256KB 测试文件（包含 A-Z 循环模式的数据）
2. 分配 3 个缓冲区（1024 + 1024 + 2048 字节）
3. 构造包含 3 个 iovec 的数组
4. 调用 `preadv(fd, iov, 3, 0)` 从偏移量 0 开始读取
5. 验证读取字节数为 4096
6. 验证所有缓冲区中的数据与源文件一致

**验证点**:
- ✅ 系统调用成功执行
- ✅ 返回正确的字节数
- ✅ 数据完整性校验通过（每个字节符合预期模式）

**预期结果**: `[PASS]` - 成功读取 4096 字节到分散的缓冲区中

#### 1.2 test_pwritev_basic - 基础向量写入

**描述**: 使用 3 个 iovec 结构体将不同模式的数据写入文件

3. 构造包含 3 个 iovec 的数组
4. 调用 `pwritev(fd, iov, 3, 0)` 写入数据
5. 执行 fsync 确保数据持久化
6. 回退到文件开头并使用普通 read() 读回数据
7. 验证读回数据的正确性

**验证点**:
- ✅ pwritev 成功写入 4096 字节
- ✅ 数据按 iovec 顺序正确拼接
- ✅ 第 0-1023 字节为 'X'

**预期结果**: `[PASS]` - 数据正确写入并可读回验证

**验证点**:
- ✅ 正确从指定偏移量开始读取

### 2. 标志测试 (test_flags)

**目标**: 验证 `preadv2`/`pwritev2` 的扩展标志位支持

**测试数量**: 7 个

**前置知识**: Linux 4.6+ 内核引入 `preadv2`/`pwritev2`，支持以下标志：
- `RWF_NOWAIT` (0x08): 非阻塞 I/O，如果需要等待则立即返回 EAGAIN
- `RWF_HIPRI` (0x01): 高优先级 I/O
- `RWF_DSYNC` (0x02): 写操作后同步数据（不含元数据）
- `RWF_SYNC` (0x04): 写操作后同步数据和元数据
- `RWF_APPEND` (0x10): 追加写入模式

#### 2.1 test_preadv2_no_flags - 无标志的 preadv2

**描述**: 使用 flags=0 调用 preadv2，验证其兼容性

**测试步骤**:
1. 创建 256KB 测试文件
2. 构造 2 个 iovec（各 1024 字节）
3. 调用 `preadv2(fd, iov, 2, 0, 0)` （flags 参数为 0）
4. 验证读取 2048 字节

**验证点**:
- ✅ preadv2 在无额外标志时正常工作
- ✅ 行为与 preadv 完全一致
- ✅ 接口向后兼容

**预期结果**: `[PASS]` - 无标志模式下正常工作

---

#### 2.2 test_preadv2_nowait - RWF_NOWAIT 非阻塞读取

**描述**: 测试 RWF_NOWAIT 标志的非阻塞 I/O 行为

**测试步骤**:
1. 创建 256KB 测试文件
2. 构造单个 iovec（4096 字节）
3. 调用 `preadv2(fd, &iov, 1, 0, RWF_NOWAIT)`
4. 根据返回值判断：
   - 如果成功读取：说明数据已在缓存中，直接返回
   - 如果 errno == EAGAIN：对于网络文件系统是正常的（需要等待 I/O）
   - 如果 errno == EOPNOTSUPP/EINVAL：当前内核或文件系统不支持此标志

**验证点**:
- ✅ 不会导致进程阻塞
- ✅ 对于 FUSE/JuiceFS 可能返回 EAGAIN（网络延迟）
- ✅ 错误处理合理

**可能结果**: `[PASS]` 或 `[SKIP]`（取决于文件系统支持程度）

---

#### 2.3 test_preadv2_hipri - RWF_HIPRI 高优先级读取

**描述**: 测试高优先级 I/O 标志的支持情况

**测试步骤**:
1. 创建 256KB 测试文件
2. 构造单个 iovec（4096 字节）
3. 调用 `preadv2(fd, &iov, 1, 0, RWF_HIPRI)`

**验证点**:
- ✅ RWF_HIPRI 标志被接受
- ✅ 数据读取完整无误
- ✅ 高优先级请求得到服务

**可能结果**: `[PASS]` 或 `[SKIP]`（某些文件系统不支持优先级调度）

---

#### 2.4 test_pwritev2_sync - RWF_SYNC 同步写入

**描述**: 测试带 RWF_SYNC 标志的原子性同步写入

**测试步骤**:
1. 创建新文件
2. 准备 4096 字节缓冲区（填充字符 'S'）
3. 调用 `pwritev2(fd, &iov, 1, 0, RWF_SYNC)`
4. 同步将文件指针移到开头，用 read() 读回数据
5. 比较原始缓冲区和读回数据

**验证点**:
- ✅ 写入完成后数据已持久化到存储介质
- ✅ 元数据和数据都同步完成
- ✅ 数据完整性校验通过
- ✅ 类似于 write() + fsync() 的效果，但更高效（一次系统调用）

**可能结果**: `[PASS]` 或 `[SKIP]`（取决于文件系统对 RWF_SYNC 的支持）

---

#### 2.5 test_pwritev2_dsync - RWF_DSYNC 数据同步写入

**描述**: 测试 RWF_DSYNC 标志（仅同步数据，不同步元数据修改时间）

**测试步骤**:
1. 创建新文件
2. 准备 4096 字节缓冲区（填充字符 'D'）
3. 调用 `pwritev2(fd, &iov, 1, 0, RWF_DSYNC)`
4. 读回数据并比较

**验证点**:
- ✅ 数据已刷新到持久存储
- ✅ 数据内容完全一致
- ✅ 性能优于 RWF_SYNC（不需要等待元数据更新）

**RWF_DSYNC vs RWF_SYNC 区别**:
- `RWF_DSYNC`: 只保证数据落盘，不保证元数据（如修改时间）立即更新
- `RWF_SYNC`: 保证数据和元数据都落盘

**可能结果**: `[PASS]` 或 `[SKIP]`

#### 2.6 test_preadv2_sync_read - 读取操作的 RWF_SYNC 标志

**描述**: 在读取操作上使用 RWF_SYNC 标志（语义较弱）

**测试步骤**:

**验证点**:
- ✅ 对于读取操作，RWF_SYNC 可能被接受或忽略
- ✅ 不会产生意外错误
- ✅ 读取功能正常完成

**说明**: RWF_SYNC 主要针对写操作设计，用于读操作时语义较弱，主要用于确保读取的是最新数据。

**可能结果**: `[PASS]` 或 `[SKIP]`

---

#### 2.7 test_pwritev2_append - RWF_APPEND 追加写入

**描述**: 测试 RWF_APPEND 标志实现原子性的追加写入

**测试步骤**:
1. 创建新文件并写入初始字符串 "INITIAL"（7 字节）
2. 执行 fsync 确保初始数据持久化
3. 准备追加数据 "APPENDED"（8 字节）
4. 调用 `pwritev2(fd, &iov, 1, -1, RWF_APPEND)`（offset 设为 -1 表示自动定位到末尾）
5. 读回整个文件内容
6. 验证内容为 "INITIALAPPENDED"（15 字节）

**验证点**:
- ✅ 数据被追加到文件末尾而非覆盖
- ✅ offset 参数为 -1 时自动定位到 EOF
- ✅ 追加操作具有原子性（并发安全）
- ✅ 最终文件内容正确拼接

**RWF_APPEND 的优势**:
- 相比 lseek 到末尾 + write，避免了竞态条件
- 在多线程/多进程环境下更安全

**可能结果**: `[PASS]` 或 `[SKIP]`

### 3. O_DIRECT 测试 (test_odirect)

**目标**: 验证 O_DIRECT 模式下 preadv/pwritev 的正确性和限制

**前置条件**: 需要 JuiceFS 挂载时启用 O_DIRECT 支持

**测试数量**: 4 个

**O_DIRECT 关键要求**:
- 缓冲区地址必须内存对齐（通常为 512 字节或 4096 字节边界）
- I/O 大小必须是对齐值的整数倍
- 传输偏移量必须对齐

#### 3.1 test_odirect_preadv_aligned - O_DIRECT 对齐读取

**描述**: 在 O_DIRECT 模式下使用内存对齐的缓冲区进行 preadv

**测试步骤**:
1. 使用 `open(path, O_RDWR | O_CREAT | O_TRUNC | O_DIRECT, 0644)` 打开文件
2. 如果打开失败且 errno 为 EINVAL/EOPNOTSUPP，说明文件系统不支持 O_DIRECT，标记为 SKIP
3. 使用 `posix_memalign()` 分配 4096 字节对齐的写入缓冲区
4. 用 `pwrite()` 写入 4096 字节数据（字符 'O'）
5. 再次使用 posix_memalign 分配对齐的读取缓冲区
6. 构造 iovec 并调用 `preadv(fd, &iov, 1, 0)` 读取
7. 比较写入和读取的缓冲区内容

**验证点**:
- ✅ O_DIRECT 模式下文件打开成功
- ✅ 对齐缓冲区的 preadv 正确执行
- ✅ 绕过页缓存直接从磁盘读取
- ✅ 数据完整性保持一致

**可能结果**: `[PASS]` 或 `[SKIP]`（如果文件系统不支持 O_DIRECT）

---

#### 3.2 test_odirect_pwritev_aligned - O_DIRECT 对齐写入

**描述**: 在 O_DIRECT 模式下使用多个对齐缓冲区进行 pwritev

**测试步骤**:
1. 以 O_DIRECT 模式创建文件
2. 使用 posix_memalign 分配两个 4096 字节的对齐缓冲区
3. 分别填充 'A' 和 'B' 字符
4. 构造 2 个 iovec 并调用 `pwritev(fd, iov, 2, 0)` 写入 8192 字节
5. 关闭文件后以普通模式重新打开（绕过 O_DIRECT）
6. 使用普通 read() 读回全部数据
7. 验证前 4096 字节为 'A'，后 4096 字节为 'B'

## 常见问题

### Q: 为什么有些测试显示 [SKIP]？

A: 这不是错误。当文件系统或内核版本不支持某个特定功能时（如 O_DIRECT、RWF_NOWAIT 等），测试会优雅地跳过并标注原因。

### Q: 如何判断测试是否全部通过？

A: 查看每个测试程序的 Summary 输出，确认 FAIL 数量为 0。run_tests.sh 会运行所有测试程序并汇总结果。

### Q: 测试失败的可能原因有哪些？

1. JuiceFS 挂载选项不正确（缺少必要的功能支持）
2. 内核版本过旧（某些 flag 不支持）
3. 权限不足（无法创建测试文件）
4. 磁盘空间不足
5. FUSE 版本兼容性问题

### Q: 可以单独运行某个测试吗？

A: 可以。每个测试程序都可以独立运行，只需传入测试目录路径即可。
